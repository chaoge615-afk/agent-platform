## 前言

上一篇讲了 MCP 协议怎么把外部工具服务接进 Agent，这篇来讲**记忆系统**。

大模型本身是无状态的，每次调用都是一张白纸。没有记忆的话，用户说"我喜欢简洁的回答"，下一轮它又给你写小作文；上次分类分错了，下次碰到同样的问题还是犯一样的错。

所以我给 Agent 设计了三层记忆：短期记忆管对话上下文，长期记忆管用户偏好和事实，反思记忆管 Agent 自己的改进洞察。三层记忆各司其职，配合 ChromaDB 的向量检索，让 Agent 真的能"越用越聪明"。

## 一、为什么 Agent 需要记忆

先说一个我实际遇到的问题。用户第一轮说"帮我查桃姐最近聊了什么话题"，Agent 走 SQL 没问题。第二轮说"她关于吵架的建议是什么"，Agent 走 RAG 也没问题。但第三轮说"结合刚才两个结果帮我总结一下"——Agent 懵了，因为它不记得前两轮聊了什么。

这就是没有记忆的后果：**每轮对话都是一次全新请求，上下文断裂**。还有一类问题更隐蔽——用户纠正了"不对，我说的是张三不是李四"，如果 Agent 没记下来，下次遇到类似歧义还是会搞混。

所以我需要三层记忆来解决不同的问题：

| 记忆层级 | 存储位置 | 作用 | 生命周期 |
|---------|---------|------|---------|
| 短期记忆 | 内存字典 | 对话上下文，多轮连贯 | 单次会话 |
| 长期记忆 | ChromaDB | 用户偏好、事实、模式 | 跨会话持久化 |
| 反思记忆 | ChromaDB | Agent 自我改进洞察 | 跨会话持久化 |

短期记忆保证"这次聊天能接上话"，长期记忆保证"下次还记得你的喜好"，反思记忆保证"犯过的错不再犯"。

## 二、三层记忆架构总览

先看一张全景图：

```
                    用户问题
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
     短期记忆      长期记忆     反思记忆
    (内存字典)   (ChromaDB)   (ChromaDB)
          │           │           │
    get_short_term    │    search_reflections
          │           │           │
          └───────────┼───────────┘
                      ▼
              classify_intent
         (对话历史 + memory_context + 问题 → LLM 分类)
                      │
          route → query → merge → 生成回答
                      │
                      ▼
                  reflect 节点
            (问答复盘 → save_reflection → ChromaDB)
                      │
              下次请求时被检索，注入分类 Prompt
```

整个闭环的关键在于：**反思记忆不是存完就完了，它会在下一次请求时被检索出来，注入到分类 Prompt 中，影响 Agent 的路由决策**。这就是"越用越聪明"的核心机制。

## 三、ChromaDB 初始化：嵌入式向量存储

### 1. PersistentClient 与 Collection

ChromaDB 的初始化在 `MemoryStore.connect()` 中完成。我选的是 `PersistentClient`（本地嵌入式模式），不需要额外起服务进程，数据直接写到本地磁盘：

```python
async def connect(self):
    """初始化 ChromaDB 嵌入式客户端"""
    os.makedirs(self.data_path, exist_ok=True)
    self._client = chromadb.PersistentClient(path=self.data_path)

    # 长期记忆 collection
    self._long_term_collection = self._client.get_or_create_collection(
        name=self.collection_name,
        metadata={"hnsw:space": "cosine"},
    )

    # 反思记忆 collection
    self._reflection_collection = self._client.get_or_create_collection(
        name="agent_reflections",
        metadata={"hnsw:space": "cosine"},
    )

    print(f"[Memory] ChromaDB 已初始化: {self.data_path}")
```

几个设计要点：`PersistentClient` 比 `HttpClient` 轻量，不需要起服务进程；`get_or_create_collection` 是幂等的，第一次创建，后续直接获取；两个 Collection 各管各的——`agent_memory` 存长期记忆，`agent_reflections` 存反思洞察。

配置项在 `Config` 里：

```python
class Config:
    CHROMA_DATA_PATH: str = os.getenv("CHROMA_DATA_PATH", "./data/chroma_db")
    MEMORY_COLLECTION: str = os.getenv("MEMORY_COLLECTION", "agent_memory")
```

### 2. Cosine 相似度

为什么选 cosine 而不是 L2（欧氏距离）或 IP（内积）？记忆检索关心的是**语义方向是否一致**，而不是向量的绝对大小。"怎么优化代码"和"代码优化方法"措辞不同但语义方向一致，cosine 能给出高分。ChromaDB 底层用 HNSW 索引做近似最近邻搜索，几千条记忆的规模下查询延迟在毫秒级。

## 四、短期记忆：内存字典 + 滑动窗口

短期记忆是最简单的一层，用一个内存字典就行：

```python
# 短期记忆：{conversation_id: [messages]}
self._short_term: dict[str, list] = {}
```

### 1. save_short_term：追加消息

```python
async def save_short_term(self, conversation_id: str, message: dict):
    """保存短期记忆（对话上下文）

    Args:
        conversation_id: 对话 ID
        message: {"role": "user"/"assistant", "content": "..."}
    """
    if conversation_id not in self._short_term:
        self._short_term[conversation_id] = []

    self._short_term[conversation_id].append({
        **message,
        "timestamp": time.time(),
    })

    if len(self._short_term[conversation_id]) > 50:
        self._short_term[conversation_id] = \
            self._short_term[conversation_id][-50:]
```

每条消息用 `**message` 展开原始字典保留 `role` 和 `content`，同时追加 `timestamp`。超过 50 条后用 `[-50:]` 切片只保留最新的。

### 2. get_short_term：取最近 N 条

```python
async def get_short_term(self, conversation_id: str, limit: int = 10) -> list:
    """获取短期记忆（最近 N 条消息）"""
    messages = self._short_term.get(conversation_id, [])
    return messages[-limit:]
```

默认取最近 10 条，`[-limit:]` 取列表末尾 N 个元素。

### 3. 50 条上限与滑动窗口

为什么限制 50 条？一是 **LLM 上下文窗口有限**，把所有历史都塞进 Prompt 既费 token 又影响质量；二是**防止内存泄漏**，这个字典在内存里，不限制的话长会话会让内存持续增长。

50 条大约覆盖 25 轮对话（一问一答算 2 条），超过后旧消息被丢弃——这就是**滑动窗口**：窗口大小固定，新消息进来，旧消息被推出。

```
msg1  msg2  msg3  ...  msg48  msg49  msg50  msg51
│                                  │           │
└──── 被丢弃 ──────────────────────┘           │
     │←───── 保留最近 50 条 ──────────────────→│
```

## 五、长期记忆：ChromaDB 语义检索

### 1. save_long_term：写入向量库

```python
async def save_long_term(self, user_id: str, memory: dict):
    """保存长期记忆

    Args:
        user_id: 用户 ID
        memory: {"content": "...", "type": "preference/fact/pattern"}
    """
    if not self._long_term_collection:
        return

    doc = memory.get("content", "")
    metadata = {
        "user_id": user_id,
        "type": memory.get("type", "general"),
        "timestamp": time.time(),
    }
    mem_id = f"mem_{uuid.uuid4().hex[:12]}"

    self._long_term_collection.add(
        documents=[doc],
        metadatas=[metadata],
        ids=[mem_id],
    )
```

几个细节值得注意：`mem_` 前缀方便调试时区分数据来源；`uuid4().hex[:12]` 在几千条规模下碰撞概率可忽略；metadata 里的 `type` 支持 `preference`（偏好）、`fact`（事实）、`pattern`（模式）三种分类。

最重要的一点：**ChromaDB 会自动做 embedding**。`add()` 接收文本，ChromaDB 内部用默认的 embedding 模型（all-MiniLM-L6-v2）把文本转成向量，不需要我们手动调 embedding API。

### 2. search_long_term：语义查询

```python
async def search_long_term(self, query: str, top_k: int = 5) -> list:
    """语义检索长期记忆

    Args:
        query: 查询文本
        top_k: 返回结果数

    Returns:
        [{"content": "...", "metadata": {...}, "distance": float}]
    """
    if not self._long_term_collection:
        return []

    if self._long_term_collection.count() == 0:
        return []

    results = self._long_term_collection.query(
        query_texts=[query],
        n_results=min(top_k, self._long_term_collection.count()),
    )

    memories = []
    if results and results["documents"]:
        for i, doc in enumerate(results["documents"][0]):
            memories.append({
                "content": doc,
                "metadata": results["metadatas"][0][i]
                    if results["metadatas"] else {},
                "distance": results["distances"][0][i]
                    if results["distances"] else None,
            })
    return memories
```

两个防御性判断值得注意：`count() == 0` 防止对空 Collection 做 query 抛异常；`min(top_k, count())` 防止 `n_results` 超过文档总数报错。返回的 `distance` 是余弦距离（1 - cosine_similarity），值越小越相关。

## 六、反思记忆：Agent 的自我改进

反思记忆是这个系统最有意思的部分。它不是存用户的输入，而是存 **Agent 自己总结的改进洞察**。

### 1. save_reflection：存储反思洞察

```python
async def save_reflection(self, reflection: dict):
    """保存反思记录

    Args:
        reflection: {"content": "...", "type": "...", "question": "..."}
    """
    if not self._reflection_collection:
        return

    doc = reflection.get("content", "")
    metadata = {
        "type": reflection.get("type", "general"),
        "question": reflection.get("question", ""),
        "timestamp": time.time(),
    }
    ref_id = f"ref_{uuid.uuid4().hex[:12]}"

    self._reflection_collection.add(
        documents=[doc],
        metadatas=[metadata],
        ids=[ref_id],
    )
```

结构和长期记忆类似，但 metadata 多了 `question`——记录触发这条洞察的原始问题。`type` 有四种：`preference`（用户偏好）、`correction`（纠正错误）、`pattern`（可复用模式）、`quality`（质量改进）。

### 2. search_reflections：检索反思

```python
async def search_reflections(self, query: str, top_k: int = 3) -> list:
    """语义检索反思记录"""
    if not self._reflection_collection:
        return []
    if self._reflection_collection.count() == 0:
        return []

    results = self._reflection_collection.query(
        query_texts=[query],
        n_results=min(top_k, self._reflection_collection.count()),
    )

    reflections = []
    if results and results["documents"]:
        for i, doc in enumerate(results["documents"][0]):
            reflections.append({
                "content": doc,
                "metadata": results["metadatas"][0][i]
                    if results["metadatas"] else {},
            })
    return reflections
```

默认 `top_k=3`，比长期记忆少——反思洞察高度浓缩，2-3 条就够了，太多反而干扰分类。

## 七、反思注入 classify_intent：闭环是怎么形成的

这一节是全文重点。前面讲了三层记忆的存储和检索，关键是**怎么用起来**。整个闭环分三步：请求入口检索反思 → 分类节点注入 Prompt → 反思节点写入洞察。

### 1. 请求入口：检索反思

在 `main.py` 的请求处理函数里，每次收到用户问题就检索相关的反思洞察：

```python
# src/main.py
reflections = await mem.search_reflections(request.question, top_k=2)
if reflections:
    memory_context = "过往改进洞察：\n" + "\n".join(
        f"- {r['content']}" for r in reflections
    )

initial_state = {
    "question": request.question,
    "conversation_id": conv_id,
    "messages": history,
    "memory_context": memory_context,
}
```

注意 `top_k=2`，只取最相关的 2 条。`memory_context` 被组装成文本，作为 `initial_state` 的字段传入 LangGraph。

### 2. 分类节点：注入 Prompt

在 `classify_intent` 节点函数里，`memory_context` 被注入到分类 Prompt 中：

```python
async def classify_intent(state: AgentState) -> dict:
    prompt_parts = [INTENT_SYSTEM_PROMPT]

    # 注入对话历史（最近 3 轮 = 6 条消息）
    messages = state.get("messages") or []
    if messages:
        recent = messages[-6:]
        history = "\n".join(
            f"{'用户' if m.get('role') == 'user' else '助手'}: "
            f"{m.get('content', '')}" for m in recent
        )
        prompt_parts.append(f"\n对话历史：\n{history}")

    # 注入反思上下文（关键行）
    memory_context = state.get("memory_context")
    if memory_context:
        prompt_parts.append(f"\n相关背景：\n{memory_context}")

    prompt_parts.append(f"\n\n用户问题：{state['question']}")
```

这样 LLM 做分类决策时，不仅看到用户问题和对话历史，还看到**过往的改进洞察**。比如之前有条反思说"用户偏好简洁回答"，LLM 分类时就能考虑这个偏好。

### 3. reflect 节点：自我改进反馈循环

回答生成之后，`reflect` 节点对这次问答做复盘。先看反思 Prompt：

```python
REFLECT_PROMPT = """你是一个 AI 助手的质量改进模块。
回答格式（JSON）：
{
    "should_save": true/false,
    "insight": "一句话总结的洞察或改进点",
    "type": "preference|correction|pattern|quality"
}
判断标准：
- should_save=true: 用户有明确偏好、纠正了错误、
  或发现了可复用的模式
- should_save=false: 普通问答，没有特别值得记录的洞察
只返回 JSON，不要其他文字。"""
```

`reflect` 节点函数调用 LLM 评估这次问答，有值得记录的洞察就写入反思记忆：

```python
async def reflect(state: AgentState) -> dict:
    question = state.get("question", "")
    answer = state.get("final_answer", "")

    if not answer or answer == "暂无结果" or "未连接" in answer:
        return {}

    messages = [
        {"role": "system", "content": REFLECT_PROMPT},
        {"role": "user",
         "content": f"问题：{question}\n回答：{answer[:500]}"},
    ]

    try:
        response = await call_llm_direct(messages)
        response = response.strip()
        # 去除 markdown 代码块包裹（```json ... ```）
        if response.startswith("```json"):
            response = response[7:]
        if response.startswith("```"):
            response = response[3:]
        if response.endswith("```"):
            response = response[:-3]

        result = json.loads(response.strip())

        if result.get("should_save") and result.get("insight"):
            mem = get_memory_store()
            await mem.save_reflection({
                "content": result["insight"],
                "type": result.get("type", "general"),
                "question": question,
            })
    except Exception:
        pass  # 非关键路径，静默忽略

    return {}
```

几个细节：`answer[:500]` 只截取前 500 字符给 LLM 评估，省 token；markdown 代码块剥离是因为 LLM 经常把 JSON 包在 \`\`\`json ... \`\`\` 里返回；`except Exception: pass` 说明反思是**非关键路径**，失败了不影响主流程。

这样就形成了完整的反馈循环。举个具体例子：

```
第 1 次：用户"帮我查桃姐的话题" → SQL 查询 → 正常 → should_save: false

第 5 次：用户纠正"不对，我说的是张三不是李四"
  → reflect 提取洞察："注意上下文中的指代关系，避免混淆人名"
  → save_reflection() 写入 ChromaDB

第 6 次：用户"张三上次说了什么？"
  → search_reflections() 检索到上次的洞察
  → 注入 classify_intent Prompt → 分类更准确
```

## 八、get_stats：记忆系统体检

最后看一下统计接口，用来监控记忆系统的运行状态：

```python
async def get_stats(self) -> dict:
    """获取记忆系统统计"""
    total_short = sum(len(msgs) for msgs in self._short_term.values())
    long_count = self._long_term_collection.count() \
        if self._long_term_collection else 0
    ref_count = self._reflection_collection.count() \
        if self._reflection_collection else 0

    return {
        "short_term_conversations": len(self._short_term),
        "short_term_messages": total_short,
        "long_term_count": long_count,
        "reflection_count": ref_count,
    }
```

返回四个指标：

| 指标 | 含义 | 典型值 |
|------|------|--------|
| `short_term_conversations` | 当前活跃的会话数 | 取决于并发用户 |
| `short_term_messages` | 所有会话的消息总数 | 每会话 ≤ 50 |
| `long_term_count` | 长期记忆条数 | 持续增长 |
| `reflection_count` | 反思洞察条数 | 持续增长，但增速慢 |

这个接口可以挂到 `/health` 或管理后台。如果 `reflection_count` 一直不涨，说明 `reflect` 节点可能没正常工作；如果 `short_term_messages` 异常大，可能有会话没被正确清理。

全局单例通过模块级变量实现——`get_memory_store()` 用 `global _store` 保证整个应用只有一个 `MemoryStore` 实例，所有节点共享同一份记忆数据。

## 总结

三层记忆系统让 Agent 从无状态的"一次性工具"变成了有记忆的"智能助手"：

- **短期记忆**用内存字典 + 50 条滑动窗口，保证多轮对话的连贯性，实现简单、响应快。
- **长期记忆**用 ChromaDB 的 cosine 向量检索，把用户偏好、事实、模式持久化，支持跨会话的语义检索。
- **反思记忆**是整个系统的灵魂——`reflect` 节点在每次问答后做复盘，提取改进洞察写入 ChromaDB；下次请求时，`search_reflections` 把相关洞察检索出来，注入到 `classify_intent` 的 Prompt 中，让 Agent 的分类决策越来越准。

三层记忆各管各的存储和检索，但通过 `memory_context` 这个字段串联起来，在工作流的分类节点汇聚，形成了一个完整的自我改进闭环。
